📗 Guía del Desarrollador

Setup inicial, master key, hash, roles y flujo técnico de instalación de ADDBOX LLC

👨‍💻 Uso interno — Desarrolladores

1. Visión general del setup inicial

Qué hace el setup y por qué existe

El setup inicial de ADDBOX LLC tiene como objetivo crear la primera cuenta administrativa del sistema, asociada al rol jefe. Este proceso se ejecuta una sola vez y queda bloqueado después de completarse.

Elementos clave del setup:

  • Generación de una master key única.
  • Almacenamiento del hash SHA-256 de la master key en la tabla instalacion.
  • Creación de la cuenta del jefe en Supabase Auth.
  • Inserción del perfil en la tabla usuarios con rol jefe.
  • Marcado de first_run = false para impedir repetir el setup.
⚠️ Importante: El flujo de setup debe considerarse una operación sensible. No debe exponerse públicamente ni dejarse accesible después de completado.

2. Generación de la master key

Ruta: /admin-dashboard/setup/dev-master-key.html

La master key se genera desde una página de uso exclusivo del desarrollador:

/admin-dashboard/setup/dev-master-key.html

Formato recomendado de la master key:

ADDBOX-OWNER-xxxxxxxx-xxxxxxxx-xxxxxxxx-xxxxxxxx

Características:

  • Debe ser aleatoria y difícil de adivinar.
  • Se muestra una sola vez en pantalla.
  • Puede descargarse como archivo .key o copiarse manualmente.
  • No se almacena en texto plano en la base de datos, solo su hash.
⚠️ La master key no es recuperable desde el sistema. Si se pierde, se debe regenerar el proceso de instalación (solo en entornos controlados).

3. Hash SHA-256 y tabla instalacion

Cómo se almacena y valida la master key

En lugar de guardar la master key en texto plano, el sistema guarda su hash SHA-256 en la tabla instalacion de Supabase.

Ejemplo conceptual:

// Pseudocódigo const masterKey = "ADDBOX-OWNER-..."; const hash = sha256(masterKey); // Insertar en tabla instalacion { id: 1, master_key_hash: hash, first_run: true, created_at: ... }

Propiedades importantes:

  • El hash SHA-256 es irreversible: no se puede obtener la clave original desde la base de datos.
  • La validación se hace comparando sha256(masterKeyIngresada) con el valor almacenado.
  • La tabla instalacion debe tener un único registro activo.
✅ Buenas prácticas: limitar el acceso a la tabla instalacion mediante RLS y roles de servicio.

4. Flujo técnico de creación del jefe

Ruta: /admin-dashboard/crear-jefe.html

Cuando el jefe accede a /admin-dashboard/crear-jefe.html y envía el formulario, el sistema ejecuta el siguiente flujo:

  1. Valida que todos los campos estén completos.
  2. Valida que la contraseña cumpla con la longitud mínima.
  3. Calcula sha256(masterKeyIngresada) y la compara con master_key_hash en instalacion.
  4. Si la master key es correcta:
    • Crea el usuario en Supabase Auth (email + contraseña).
    • Inserta el perfil en la tabla usuarios con rol jefe.
    • Actualiza first_run = false en instalacion.
  5. Redirige al login: /admin-dashboard/inicio-de-sesion.html.
Supabase Auth auth
- Crea el usuario con email y contraseña.
- Maneja login, logout y recuperación de contraseña.
Tabla usuarios perfil
- Guarda nombre, rol, email, metadatos.
- Se usa para permisos y vistas internas.

5. Campo first_run y bloqueo del setup

Control de una sola ejecución

El campo first_run en la tabla instalacion indica si el sistema ya fue configurado.

  • first_run = true → el setup aún no se ha completado.
  • first_run = false → el setup ya fue ejecutado y debe bloquearse.

El frontend y/o backend deben:

  • Impedir acceso a crear-jefe.html si first_run = false.
  • Redirigir a login o dashboard si alguien intenta acceder al setup después de completado.
⚠️ Nunca permitas que first_run vuelva a true en producción, salvo en un proceso controlado de reinstalación.

6. Validación del rol jefe

Permisos y acceso al dashboard

El rol jefe es el rol con acceso total al sistema. Algunas consideraciones:

  • El perfil del jefe en la tabla usuarios debe tener rol = 'jefe'.
  • El dashboard principal debe validar el rol antes de mostrar módulos sensibles.
  • Las políticas RLS en Supabase deben considerar el rol para permitir o denegar operaciones.

Ejemplo conceptual de perfil:

{ id: "...", auth_user_id: "...", nombre: "Carlos Rodríguez", email: "carlos@addbox.com", rol: "jefe", creado_en: "2026-05-19T..." }

7. Errores comunes y debugging

Checklist rápido para desarrolladores

Durante el setup pueden aparecer errores. Algunos puntos a revisar:

  • "Master key incorrecta" → verificar que el hash se esté calculando con el mismo algoritmo y encoding.
  • "User already registered" → el email ya existe en Supabase Auth; eliminarlo desde el dashboard de Supabase o usar otro correo.
  • "Error al guardar perfil" → revisar RLS y estructura de la tabla usuarios.
  • "Error al verificar instalación" → revisar conexión a Supabase, URL y claves de servicio.
✅ Recomendación: mantener un entorno de pruebas donde se pueda resetear la tabla instalacion y los usuarios sin afectar producción.

8. Buenas prácticas para el setup

Recomendaciones para entornos reales
  • Restringir el acceso a /setup/dev-master-key.html solo a entornos internos o protegidos.
  • No exponer la master key en logs, consola o herramientas de monitoreo.
  • Usar HTTPS en todo el flujo de setup.
  • Documentar quién tiene acceso a la master key y cómo se entrega al jefe.
  • Verificar que first_run se actualiza correctamente después del setup.
